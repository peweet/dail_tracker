# API Provenance Review

**Status:** Design doc — decision-first, no code yet. Execute the build at an ETL/feature
plateau (see `[[feedback_refactor_timing]]`).
**Date:** 2026-06-06 (rev 2 — separates provenance from freshness)
**Scope:** the FastAPI layer (`api/`) and the core it serves (`dail_tracker_core/`).
**Why:** provenance — the resources that let a user validate where a number came from — is
unevenly applied across the API. This doc maps the current state and proposes a
registry-by-reference model that makes provenance *more* complete while making payloads *lighter*.

---

## 0. Two concepts this review keeps separate

The first draft of this review muddied **provenance** with **freshness**. They are orthogonal
axes and only one of them is in scope here.

| | **Provenance** *(in scope)* | **Freshness** *(out of scope)* |
|---|---|---|
| Answers | "Where did this come from? How do I verify this figure against the authoritative original?" | "How old is the data we hold? Did the pipeline run?" |
| User need | Trust / verifiability | Currency / operational health |
| Artifact | `sources{}` links, `source_registry.generated.json`, licence/attribution | `data/_meta/freshness.json` |
| Workstream | **this review** | already owned by `[[project_freshness_architecture]]` (Unit A) |
| API home | `/v1/sources` + per-record citation links | `/v1/health` or a future `/v1/status` — **not** `/v1/sources` |

A source is no fresher for being well-cited, and data can be perfectly cited yet a year stale.
**This review is about provenance only.** Where freshness intersects (e.g. the empty
`generated_at` slot on the envelope), it is flagged as belonging to the freshness workstream, not
folded into the provenance design. See §6 for the explicit hand-off.

Within provenance there are two **layers** — both in scope, both legitimate:
- **(A) Source-level attribution** — "this resource draws on the Oireachtas Open Data API / the
  lobbying.ie register / SIPO." Publisher, licence, landing URL. Roughly one entry per upstream.
- **(B) Per-record citation** — the deep link a user clicks to see *this specific* bill PDF or
  *this specific* division record. One (or a few) per record.

The error in rev 1 was treating freshness as if it were a third provenance layer. It isn't.

---

## 1. The problem in one sentence

Provenance on the API is **bimodal**: 2 of 11 domain endpoints expose source links, the other 9
expose nothing — and the inventory that would fix it (`source_registry.generated.json`, 107
sources) already exists but is never served.

## 2. Current state (audited)

### 2.1 Per-endpoint coverage

| Endpoint | Provenance today | Mechanism |
|---|---|---|
| `GET /` | Global `licence` + `attribution` | hardcoded in [main.py:62-64](../api/main.py#L62-L64) |
| `GET /v1/catalog` | Global licence/attribution + a *promise* of "per-resource provenance" that **does not exist** | [catalog.py:89-94](../api/routers/catalog.py#L89-L94) |
| `GET /v1/health` | none (liveness only) | — |
| `GET /v1/members` | ❌ none | `serialize.envelope` |
| `GET /v1/members/{code}/dossier` | ❌ none — the **flagship** record, fusing ~8 sources, carries zero provenance | `MemberDossier` has no `sources` field |
| `GET /v1/legislation` | ❌ none | `serialize.envelope` |
| `GET /v1/legislation/{bill_id}` | ✅ `sources{}` — 3 URLs (bill page, official PDF, memo) | [legislation_sources.sql](../sql_views/legislation_sources.sql) |
| `GET /v1/statutory-instruments` | ❌ none | `serialize.envelope` |
| `GET /v1/votes` | ❌ none | `serialize.envelope` |
| `GET /v1/votes/{vote_id}` | ✅ `sources{}` — 1 URL + label | [vote_sources.sql](../sql_views/vote_sources.sql) |
| `GET /v1/payments` | ❌ none | `serialize.envelope` |
| `GET /v1/lobbying/organisations` | ❌ none | `serialize.envelope` |
| `GET /v1/lobbying/revolving-door` | ❌ none | `serialize.envelope` |

Both ✅ rows are **layer-B** per-record citation. **No endpoint exposes layer-A** source-level
attribution beyond the one global licence line.

### 2.2 The two existing `sources{}` schemas already diverge

Both dossiers expose a `sources` object, but the shapes don't match:

- `v_legislation_sources` → `bill_id, oireachtas_url, legislation_url(null), official_pdf_url,
  source_url(null), source_document_url, bill_no, bill_year`
- `v_vote_sources` → `vote_id, source_url, source_document_url(null), official_pdf_url(null),
  legislation_url(null), source_label`

Same intent, different column names (`oireachtas_url` vs `source_url`), different null padding,
one carries a human `source_label` and the other doesn't. Any third endpoint added today would
invent a third shape.

### 2.3 The provenance inventory already exists — and is never served

**`data/_meta/source_registry.generated.json`** — 107 upstream sources, generated by
`tools/build_source_registry.py` from in-code extractor configs. Per source:
`source_id, group, owner_module, name, listing_url, direct_files[], grain, privacy_risk,
status, pollable, caveat, stale_after_days, …`. **It already has a `privacy_risk` field** — the
hook needed for a safe public allow-list (§5).

But it is **finance/PDF-weighted**: it covers AFS, local-authority payments, HSE/Tusla, charities,
and the Oireachtas *PDF* sources (attendance, interests, payments). It does **not** inventory the
core parliamentary API sources (`api.oireachtas.ie` for members/bills/divisions/questions, the
Statute Book, the lobbying.ie register). Those are implicit in the code, not registered — so the
curated allow-list will need them hand-added (§5).

### 2.4 (Freshness — noted only to scope it out)

`data/_meta/freshness.json` exists and is rich (per-dataset age/status), and the envelope has an
empty `generated_at` slot ([serialize.py:92-110](../dail_tracker_core/serialize.py#L92-L110)) that
would naturally carry it. **This is the freshness concern, not provenance** — see §6 for the
hand-off to `[[project_freshness_architecture]]`. It is listed here only so a reader doesn't
mistake its absence for an oversight.

### 2.5 For contrast: the Streamlit side already solved provenance

The UI uses **registry-by-reference**, not inline citation: one `provenance_expander()` in
[source_pdfs.py](../utility/ui/source_pdfs.py) on every page, fed by central URL registries
(`INTERESTS`, `ATTENDANCE`, `PAYMENTS`) and `source_link_html()` link builders. Provenance is a
first-class, consistent element — the API just never adopted the same pattern.

## 3. The design tension (the real decision)

> "There are many discrete sources and it's cluttering up the app; however clear resources to
> help users validate what they look like is very important."

These conflict **only if provenance is inline and duplicated** — i.e. if every record embeds a
full source blob (publisher + URL + licence + caveat). Across thousands of records sharing the
same handful of upstream sources, that bloats every payload and repeats the same Oireachtas
attribution endlessly.

They **stop conflicting** under registry-by-reference: describe each source **once** (layer A),
and have records point to it by id, while keeping only the genuinely per-record citation link
inline (layer B). This is exactly the Streamlit pattern (§2.5).

---

## 4. Proposed model: registry-by-reference

### 4.1 One canonical source endpoint — `GET /v1/sources` (+ `/v1/sources/{id}`)  *(layer A)*

The single validation resource. Each entry, curated for public exposure:

```jsonc
{
  "source_id": "oireachtas_divisions",
  "name": "Houses of the Oireachtas — Division records",
  "publisher": "Houses of the Oireachtas",
  "landing_url": "https://www.oireachtas.ie/...",
  "api_url": "https://api.oireachtas.ie/...",   // when applicable
  "licence": "CC-BY-4.0",
  "grain": "division",
  "caveat": null
}
```

Served from a **curated allow-list** (§5). Built once → zero clutter on data payloads.
**No freshness field here** — that is `/v1/health`'s job (§6).

### 4.2 Lightweight references + kept citation links on records  *(layers A + B)*

Each record / list envelope carries:
- `source_ids: ["oireachtas_divisions"]` — small string array; resolve to `/v1/sources` (layer A).
- `source_links` — the specific document/page URLs for *this* record, kept inline (layer B).
  These are not clutter; they *are* the citation.

This retires the divergent `sources{}` shapes (§2.2) in favour of one shape used everywhere.

### 4.3 Make `/v1/catalog`'s promise true  *(layer A)*

Each catalog resource gains a `source_ids` list. The catalog becomes the index; `/v1/sources` is
the detail. The line "see per-resource provenance" stops being a lie.

### 4.4 Unify the provenance shape in core

Define one provenance shape in core (a small Pydantic model + a single SQL contract) and migrate
`v_legislation_sources` / `v_vote_sources` onto it. New endpoints reuse it instead of inventing a
third schema. Add `source_ids` + `source_links` to the member / payments / lobbying / SI records.

**Net effect:** provenance goes from 2/11 → 11/11 traceable, *and* payloads get lighter, *and*
there is one place to maintain it. No freshness work is entangled in any of this.

## 5. The one hard part: the public-source allow-list (PII)

[catalog.py:1-7](../api/routers/catalog.py#L1-L7) is explicit — and `[[feedback_personal_insolvency_privacy]]`
reinforces — that some sources carry PII (SIPO donor home addresses, personal insolvency naming
private citizens). **`/v1/sources` must be a curated allow-list of public sources, never an
auto-dump of `source_registry.generated.json`.**

Curation rules (proposed):
- **Exclude** any source whose `privacy_risk` is set, and anything feeding a quarantined/PII
  column. The catalog is already a deliberate allow-list — extend the same discipline to sources.
- **Add the missing headline sources** the generated registry omits (§2.3): the Oireachtas Open
  Data API (members/bills/divisions/questions), the Statute Book, the lobbying.ie register.
- Treat the curated list as **source-of-truth config**, reviewed like the catalog.

This curation is the bulk of the real work; the plumbing (endpoint + record fields) is small.

---

## 6. Explicit hand-off: freshness is a separate feature

Freshness/currency is **out of scope** for this review and belongs to
`[[project_freshness_architecture]]` (Unit A). If/when the API should expose it:

- It is fed by `data/_meta/freshness.json` (per-dataset age/status), **not** by `/v1/sources`.
- Its natural home is `GET /v1/health` (extend the liveness probe) or a new `GET /v1/status`, plus
  the existing empty `generated_at`/`mart_version` slot on `serialize.envelope` for list responses.
- It should be designed and shipped on its own, so the two concerns don't recontaminate each other.

This section exists so the freshness work is *acknowledged and parked*, not silently dropped.

## 7. Phased plan (provenance only)

Each phase is independently shippable and read-only (serving-layer additions over an existing
inventory — no pipeline/ETL change).

- **Phase 0 — this doc + curated allow-list config.** Hand-author the public-source allow-list
  (headline parliamentary sources + the safe subset of the generated registry). No endpoint yet.
- **Phase 1 — `GET /v1/sources` (+ `/{id}`).** Serve the allow-list. The single validation
  resource lands. Highest value, lowest risk.
- **Phase 2 — per-resource `source_ids` in `/v1/catalog`.** Make the existing promise true.
- **Phase 3 — unify `sources{}` + add `source_ids`/`source_links` to all records.** One core
  provenance shape; migrate the legislation/vote views; extend to member/payments/lobbying/SI.

MVP = **Phases 0–2**: one place to validate every resource, without touching any data payload
shape. (Freshness phases live in the freshness workstream, §6 — deliberately not numbered here.)

## 8. Open questions for the build kickoff

1. **Allow-list home:** a hand-authored `data/_meta/api_sources.json` (reviewed like the
   catalog), or extend `build_source_registry.py` to emit a `public: true` flag and filter on it?
   (Leaning hand-authored — the headline parliamentary sources aren't in the generated registry
   at all, and the curation needs human review.)
2. **`source_ids` vocabulary:** reuse the registry's `source_id` strings, or mint API-facing ids
   that map *to* registry ids where one exists (so the public contract isn't hostage to internal
   extractor renames)? (Leaning the latter.)
3. **Does the member dossier list all 8 contributing `source_ids`, or just the headline ones?**
   (Leaning: list all — they're short strings — and keep `source_links` to the few genuinely
   per-record documents.)

---

## 9. Recommendation

**Review = yes; sprinkle-more-`sources{}` = no; and keep freshness out of it.** Adopt
registry-by-reference (§4), gated by a curated public-source allow-list (§5), with freshness
explicitly handed off to its own workstream (§6). It resolves the stated tension directly: one
validation resource, lighter payloads, complete coverage, one place to maintain. Build at a
plateau; start with Phases 0–2.
